\chapter{Metode Penelitian}

\section{Alat dan Bahan}

Penelitian ini membutuhkan perangkat keras dan perangkat lunak yang spesifik untuk membangun lingkungan \textit{testbed} blockchain yang terisolasi dan terkendali.

\subsection{Alat Penelitian}
Perangkat keras dan infrastruktur yang digunakan dalam penelitian ini meliputi:
\begin{itemize}
    \item \textbf{Laptop Pengembangan}: Lenovo Ideapad Gaming 3 (AMD Ryzen 5 4600H, 16GB RAM, Ubuntu 24.04 LTS). Digunakan untuk pengembangan skrip, manajemen klaster, dan analisis data.
    \item \textbf{Cloud VPS (Virtual Private Server)}:
    \begin{itemize}
        \item 1x \textbf{VPS Control Plane} (Contabo Cloud VPS 30): Processor \textbf{AMD EPYC}, 8 vCPU, 24 GB RAM, 200 GB NVMe. Berfungsi sebagai \textit{host} untuk orkestrator Caliper, Prometheus, Grafana, dan PostgreSQL.
        \item 5x \textbf{VPS Nodes} (Contabo Cloud VPS 20): Spesifikasi dasar 6 vCPU, 12 GB RAM, 200 GB SSD. Infrastruktur ini bersifat heterogen yang terdiri dari 2 node berbasis \textbf{AMD EPYC} dan 3 node berbasis \textbf{Intel Broadwell}. Node ini berfungsi sebagai \textit{worker nodes} (System Under Test) yang menjalankan protokol konsensus PoA dan PoS.
    \end{itemize}
\end{itemize}

Perangkat lunak yang digunakan terdiri dari:
\begin{itemize}
    \item \textbf{Docker \& Docker Compose}: Platform kontainerisasi untuk isolasi lingkungan node.
    \item \textbf{Kubernetes (k3s)}: Orkestrator untuk manajemen kluster \textit{multi-node}.
    \item \textbf{Go-Ethereum (Geth)} v1.13.15 (PoA) \& v1.16.7 (PoS): Klien eksekusi blockchain Ethereum.
    \item \textbf{Prysm} v6.0.4: Klien konsensus (Beacon Chain) untuk PoS.
    \item \textbf{Clef}: Manajemen akun dan penandatanganan transaksi untuk PoA.
    \item \textbf{Hyperledger Caliper} v0.6.0: \textit{Workload generator} untuk \textit{benchmarking}.
    \item \textbf{Prometheus, Grafana, Pushgateway, Node-Exporter}: Stack monitoring untuk metrik sistem dan blockchain.
    \item \textbf{PostgreSQL} v17: Basis data untuk menyimpan hasil eksperimen mentah.
    \item \textbf{Kurtosis} v1.13.2: Alat orkestrasi untuk \textit{deployment} jaringan PoS yang kompleks.
\end{itemize}

\subsection{Bahan Penelitian}
Bahan yang digunakan dalam penelitian ini meliputi:
\begin{itemize}
    \item \textbf{Dataset Transaksi Sintetis}: Beban kerja yang dihasilkan secara deterministik oleh Caliper.
    \item \textbf{Smart Contract DChain}: Kontrak pintar riil yang digunakan di DChain (CertificateContract, SertifikatLAM) untuk simulasi kasus penggunaan nyata.
    \item \textbf{Image Kontainer}: Docker image resmi untuk Geth, Prysm, dan tools pendukung.
\end{itemize}

\section{Metode yang Digunakan}

\subsection{Pendekatan Penelitian}
Penelitian ini menggunakan pendekatan \textbf{kuantitatif eksperimental} dengan metode \textit{benchmarking} terkomputerisasi. Fokus utama adalah membandingkan kinerja mekanisme konsensus PoA dan PoS pada platform DChain melalui serangkaian uji beban terkontrol. Pendekatan ini dipilih karena mampu memberikan data empiris yang objektif mengenai metrik performa sistem terdistribusi. Eksperimen dilakukan dalam lingkungan \textit{testbed} terisolasi untuk meminimalkan gangguan eksternal.

\subsection{Desain Eksperimen}

Desain eksperimen dirancang untuk menjawab pertanyaan penelitian mengenai perbandingan \textit{throughput}, latensi, penggunaan sumber daya, dan stabilitas.

\textbf{Variabel Kontrol}:
Untuk memastikan validitas internal, parameter berikut dijaga konstan:
\begin{itemize}
    \item \textbf{Block Time}: 5 detik (PoA) dan 12 detik (PoS - Slot Time standar Ethereum 2.0). 
    \item \textbf{Gas Limit}: 30.000.000 gas per blok.
    \item \textbf{Spesifikasi VPS}: Identik untuk semua trial.
    \item \textbf{Durasi Pemanasan (\textit{Pre-warm})}: Dilakukan sebelum setiap sesi pengukuran untuk menghilangkan efek \textit{cold start}.
    \item \textbf{Mempool Cleaning}: Memastikan antrean transaksi kosong sebelum memulai skenario baru.
\end{itemize}

\subsection{Skenario Benchmarking}
Penelitian ini menerapkan lima skenario utama untuk menguji berbagai aspek kinerja:

\begin{enumerate}
    \item \textbf{Throughput Capacity (Step Load)}:
    \begin{itemize}
        \item Tujuan: Mencari titik jenuh (\textit{knee point}) sistem.
        \item Metode: Meningkatkan beban TPS secara bertahap (5, 10, ..., 100 TPS) hingga latensi melonjak drastis.
    \end{itemize}

    \item \textbf{Fixed Load (Baseline)}:
    \begin{itemize}
        \item Tujuan: Mengukur kapasitas maksimum teoretis tanpa jeda pengiriman.
        \item Metode: Mempertahankan antrean transaksi tetap (\textit{backlog} 20 tx) selama 180 detik.
    \end{itemize}

    \item \textbf{Worker Scalability}:
    \begin{itemize}
        \item Tujuan: Menguji efisiensi penanganan klien konkuren.
        \item Metode: Menjalankan beban TPS optimal dengan variasi 1, 3, dan 5 \textit{worker}.
    \end{itemize}

    \item \textbf{Read Intensive}:
    \begin{itemize}
        \item Tujuan: Mengukur kinerja operasi baca (RPC) yang dominan pada aplikasi nyata.
        \item Metode: Beban baca tinggi (200, 400, 600 TPS) pada data yang sudah ada.
    \end{itemize}
    
    \item \textbf{Functional Lifecycle \& Stability}:
    \begin{itemize}
        \item Tujuan: Menguji fungsi penuh (Mint-Read-Burn) dan stabilitas jangka panjang.
        \item Metode: \textit{Soak Test} selama 15 menit pada 60\% beban optimal.
    \end{itemize}
\end{enumerate}

\subsection{Konfigurasi Testbed}
\subsubsection{Jaringan PoA (Proof of Authority)}
Menggunakan protokol Clique dengan Geth + Clef. Topologi didistribusikan ke 5 VPS menggunakan Kubernetes (Host Network). Setiap \textit{signer} (validator) diamankan dengan Clef yang menyimpan \textit{private key} secara terekstraksi.

\subsubsection{Jaringan PoS (Proof of Stake)}
Menggunakan protokol Gasper (LMD-GHOST + Casper FFG) dengan arsitektur Geth (Execution Layer) dan Prysm (Consensus Layer). 

Orkestrasi dilakukan menggunakan Kurtosis untuk mengelola kompleksitas \textit{peering} antara EL dan CL. Setiap node validator menjalankan 22 validator keys (total 110 validator untuk 5 node) untuk mensimulasikan beban jaringan nyata.

Visualisasi arsitektur eksperimen secara keseluruhan dapat dilihat pada Gambar \ref{fig:experimental_topology}. Topologi ini menerapkan skema \textbf{10 Node} yang terdiri dari:
\begin{enumerate}
    \item \textbf{Control Plane}: Bertindak sebagai orkestrator yang mengirimkan beban transaksi (\textit{RPC Load}) ke node observer menggunakan Hyperledger Caliper.
    \item \textbf{Validator Nodes (Signers/Proposers)}: 5 node utama (V1--V5) yang memiliki hak untuk memvalidasi transaksi dan menyusun blok baru. Dalam konsensus PoA, node-node ini terhubung dalam topologi cincin (\textit{ring}) untuk rotasi \textit{proposer} secara \textit{Round-Robin} untuk PoA dan acak untuk PoS.
    \item \textbf{Observer Nodes (Non-Validators)}: 5 node tambahan (O1--O5) yang tidak memproduksi blok tetapi berpartisipasi dalam sinkronisasi data melalui protokol \textit{gossip} (P2P). Node ini mensimulasikan institusi anggota konsorsium yang hanya memegang salinan \textit{ledger} tanpa hak validasi. Node ini berfungsi sebagai pintu masuk (\textit{gateway}) bagi transaksi yang dikirimkan oleh Control Plane. Meskipun tidak memiliki validasi blok, Observer Node bertanggung jawab untuk menerima beban transaksi injeksi dari Caliper dan mempropagasi transaksi tersebut ke seluruh jaringan (termasuk ke Validator Node) agar dapat diproses dan dimasukkan ke dalam blok baru.
\end{enumerate}

Rincian pemetaan alokasi \textit{node} ke infrastruktur VPS beserta alamat IP yang digunakan untuk setiap topologi dapat dilihat secara lengkap pada \textbf{Lampiran D}.

\subsection{Monitoring dan Pengumpulan Data}
Sistem monitoring dibangun menggunakan Prometheus yang mengumpulkan metrik dari tiga sumber:
\begin{enumerate}
    \item \textbf{Geth/Prysm}: Metrik internal blockchain (peer count, block height, tx pool).
    \item \textbf{Node Exporter/cAdvisor}: Metrik infrastruktur (CPU, RAM, Network container).
    \item \textbf{Hyperledger Caliper}: Metrik transaksi (TPS, Latency, Errors) yang dikirim via Pushgateway.
\end{enumerate}
Data disimpan di PostgreSQL untuk analisis lebih lanjut.

\section{Alur Penelitian}

Penelitian ini dilaksanakan dalam beberapa tahapan sistematis, mulai dari studi pendahuluan hingga penarikan kesimpulan.

\begin{enumerate}
    \item \textbf{Fase 1: Studi Pendahuluan}. Identifikasi masalah, studi literatur mengenai PoA dan PoS, serta analisis arsitektur DChain \textit{existing}.
    \item \textbf{Fase 2: Pengembangan Lingkungan PoA}. Replikasi jaringan PoA DChain ke lingkungan lokal dan VPS, serta pembuatan skrip otomasi.
    \item \textbf{Fase 3: Pengembangan Lingkungan PoS}. Studi implementasi PoS Ethereum, replikasi menggunakan Kurtosis, dan validasi konektivitas.
    \item \textbf{Fase 4: Pengembangan Sistem Benchmarking}. Konfigurasi Hyperledger Caliper, pengembangan \textit{workload script}, dan integrasi sistem monitoring (Prometheus/Grafana).
    \item \textbf{Fase 5: Eksekusi dan Analisis}. Pelaksanaan eksperimen pada infrastruktur \textit{cloud}, pengumpulan data metrik, analisis statistik, dan penyusunan laporan.
\end{enumerate}

\section{Etika, Masalah, dan Keterbatasan Penelitian}

\subsection{Etika Penelitian}
Penelitian ini menekankan pada integritas ilmiah dan transparansi. Seluruh kode konfigurasi, skrip \textit{benchmark}, dan data hasil eksperimen dibuka (\textit{open source}) untuk mendukung reproduksibilitas. Penggunaan infrastruktur \textit{cloud} dilakukan secara terisolasi (Private Testnet) agar tidak mengganggu jaringan publik.

\subsection{Masalah Potensial}
\begin{itemize}
    \item \textbf{Variabilitas Jaringan}: Penggunaan VPS publik rentan terhadap fluktuasi latensi (\textit{network jitter}). Mitigasi dilakukan dengan teknik statistik robust (Median) dan pengulangan eksperimen ($N=5$).
    \item \textbf{Saturasi Sumber Daya}: Beban tinggi dapat menyebabkan \textit{throttling}. Mitigasi dilakukan dengan jeda \textit{cooldown} 10 menit antar trial.
\end{itemize}

\subsection{Keterbatasan Penelitian}
\begin{itemize}
    \item \textbf{Skala Jaringan}: Dibatasi maksimal 10 node, belum mencerminkan skala ribuan node publik.
    \item \textbf{Beban Sintetis}: Pola transaksi Caliper bersifat deterministik, mungkin berbeda dengan trafik organik pengguna asli.
    \item \textbf{Spesifikasi Hardware}: Hasil performa terikat pada spesifikasi VPS kelas menengah yang digunakan, bukan server \textit{enterprise}.
    \item \textbf{Lingkungan Terisolasi}: Tidak mencakup aspek serangan keamanan (\textit{slashing}, \textit{double spending}) atau insentif ekonomi riil.
\end{itemize}
